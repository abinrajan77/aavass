# Spec: Module 5 — Reporting, Flat Owner Self-Service Portal & Notifications (v1.0 manual)

> Maps to PRD milestone M4 (+ v1.0 slice of M5, §8.1). Read `../00-architecture-and-standards.md` and `../06-cloud-devops.md` first — this spec only adds module-specific decisions.

## Problem

Tower admins have no consolidated way to see collection performance, outstanding dues, and expenditure against income without manually cross-referencing Module 3 (billing) and Module 4 (expenditure) data — and no export they can hand to the association at a general body meeting (PRD §6.7). Flat owners have no self-service visibility into their own dues, payments, receipts, or their tower's expenditures (PRD §6.6, §9), forcing admins to answer routine "have I paid?" / "what was this money spent on?" questions one-by-one. And when a due is generated, goes overdue, or is paid, admins have no ready-made message text to send tenants/owners via WhatsApp/SMS — they draft it from scratch each time (PRD §8.1).

This module closes all three gaps: five fixed reports with PDF/CSV export, a scoped owner dashboard, and system-drafted (not auto-sent) notification text.

## Non-goals

- **No automated SMS/WhatsApp dispatch.** v1.0 only prepares message text for admin to view/copy and send via their own channel. Automated send, opt-out preferences, and delivery tracking are PRD §8.2 (v1.1), explicitly out of scope for this module and this document.
- **No cross-tower consolidated reporting for owners.** An owner with flats in multiple towers gets a simple per-tower context switch (see PRD §6.6) — not a combined multi-tower financial summary. Complex-level consolidated financials are a PRD §3.2 non-goal for the whole product, not just this module.
- **No configurable/custom report builder.** Only the 5 fixed report types in PRD §6.7 exist. No user-defined columns, filters beyond period/date-range, or saved report definitions.
- **No new financial business logic.** This module does not compute due amounts, payment status, formula application, or expenditure categorization — it reads and aggregates records already produced by Module 3 (`MaintenanceDue`, `Payment`, `Receipt`) and Module 4 (`SpecialCollectionDue`, `Expenditure`). If a number looks wrong, the bug is upstream unless it's in this module's aggregation/join logic specifically.
- **No owner write access beyond PRD §6.6 scope.** Owners can edit their own contact details, their flat's tenant info, and occupancy status (already specified as Module 2 write paths this module's frontend surfaces) — not billing formulas, payments, or expenditures.
- **No mobile optimization** — desktop/tablet only, per PRD §7.

## Edge cases

- **Owner with flats in 3 different towers switches context.** The owner portal must show a tower switcher; switching context must fully re-scope every subsequent read (dues, receipts, expenditures, tenant history) to the newly selected tower/flat — no residual data from the prior tower rendered stale in a cached list.
- **Report requested for a period with zero underlying records** (e.g., a billing cycle with no dues yet, a month with no expenditures at all). This is a valid empty state, not an error: return `200` with an empty `items` array and zeroed totals, and the frontend renders an explicit "no data for this period" message — never a 404/500 or a blank table that looks broken.
- **Export request resolves to more than 5000 rows.** Per `00-architecture-and-standards.md` §4, this must not block the request thread. The API returns `202 Accepted` with a `job_id`; the actual file is generated by a worker and placed in S3 (see `cloud.md`); the frontend polls for completion.
- **Tenant-occupied flat's notification event.** Per PRD §8.1, when the current resident is a tenant, the system must prepare **two** message variants: one addressed to the tenant (the responsible payer) and a separate copy addressed to the owner, so the admin can inform both. For an owner-occupied flat, only one variant exists.
- **Owner views tenant history for a flat they no longer own (post-sale).** Decision: the owner-portal scoping is based on *current, active* `FlatOwnership` records only (per Module 2). Once an ownership record's `date_to` is set (sale recorded), that former owner loses portal access to that flat entirely, including its tenant history — they see it fall out of their `/owners/me/flats-summary` list (and Module 2's simpler `/api/v1/me/flats`) on their next login. The tenant history itself is **not deleted**; it remains fully visible to the tower admin and to the *new* owner (per PRD §6.2.3 "system maintains full tenant history... visible to both the flat owner... and the tower admin"), preserving the audit trail without granting stale access to someone no longer party to the flat.
- **Co-owned flat.** Multiple owners on one flat each see the same scoped dashboard for that flat independently (no "primary owner only" restriction on portal *access* — only receipts/dues are named to the primary owner per PRD §5.3).
- **Report period spans a formula version change** (Module 3 versions the maintenance formula). The collection report must reflect the formula/amount actually recorded on each due at generation time, not today's formula — this module reads the stored due amount, never recomputes it.
- **Outstanding dues report requested when grace period has just changed.** Per PRD §6.3.2, grace period changes apply to future cycles only; the days-overdue calculation for existing dues must use the grace period value stored/effective at that due's generation, not the tower's current setting.
- **Notification preview requested for a due with no resident assigned** (should not happen per Module 3/4 invariants, but defensively) — return a `422` with a clear error rather than a template rendered with blank placeholders.

## Acceptance criteria

1. GIVEN a completed billing cycle with a mix of paid/pending/overdue dues, WHEN the admin requests the Monthly Collection Report for that cycle, THEN the response lists every due in that cycle with flat number, owner name(s), resident type, amount due, status, and (if paid) payment date/mode/receipt number, and the sum of "paid" amounts equals the sum of `Payment.amount_received` for that cycle in Module 3.
2. GIVEN flats with overdue maintenance and/or special collection dues, WHEN the admin requests the Outstanding Dues Report, THEN only dues currently in `Overdue` status are returned, each with `days_overdue` computed from `due_date + grace_period_at_generation` to the report's as-of date (defaulting to today).
3. GIVEN expenditure records across categories for a period, WHEN the admin requests the Expenditure Report for that period, THEN the response includes the itemised list plus a category-wise subtotal, and the grand total equals the sum of `Expenditure.amount` in that period from Module 4.
4. GIVEN a month with both maintenance/special-collection payments recorded and expenditures recorded, WHEN the admin requests the Collection vs Expenditure Summary for that month (or a financial year), THEN the response shows total collected, total expenditure, and net, with total collected counting each `Payment` exactly once (no double count across maintenance and special-collection sources).
5. GIVEN a tower with current and past tenants across multiple flats, WHEN the admin requests the Tenant Register, THEN all tenants (current and historical) across all flats in the tower are returned, ordered by flat number then lease start date, each flagged `current: true|false`.
6. GIVEN any of the 5 reports, WHEN requested with `format=pdf` or `format=csv`, THEN the response is the rendered file (synchronous, <10s) for ≤5000 rows, or a `202` + `job_id` for >5000 rows.
7. GIVEN a flat owner user authenticated and scoped to Tower A only, WHEN they call any owner-portal endpoint, THEN they never receive data for a flat/tower they do not currently own — attempting `flat_id`/`tower_id` outside their active ownership returns `403`.
8. GIVEN an owner with active flats in Tower A and Tower B, WHEN they open the owner portal, THEN `GET /api/v1/owners/me/flats-summary` returns both flats grouped by tower, and switching the Command-palette context to Tower B re-scopes the dashboard to Tower B's flat only.
9. GIVEN a `due_generated`, `overdue_reminder`, or `payment_confirmed` event for a tenant-occupied flat, WHEN the admin requests the notification preview for that due, THEN the response contains two message objects (`recipient: tenant`, `recipient: owner`) with placeholders filled from the due/flat/owner/tenant data; for an owner-occupied flat, exactly one message object (`recipient: owner`) is returned.
10. GIVEN no automated dispatch capability exists in v1.0, WHEN the notification preview is returned, THEN the API response contains no delivery status field and no send action is exposed anywhere in this module's UI or API.
